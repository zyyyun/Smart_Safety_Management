## Problem Statement

현재 시스템 (Phase 8 검증 완료):
```
Drift X3 카메라 ─RTSP→ PC (같은 LAN, scheduler.py)
                        ↓ YOLO @ RTX 3060 Ti
                        ↓ Supabase
                        ↓ FCM → 모바일 알림
latency: 3.16s (camera_id=3, person, event_id=46, Phase 8 RTSP-02)
```

가설 시나리오: **6월 검단·포천 현장에 PC 가 없다**. 모바일 앱이 카메라 RTSP 영상을
받아 본부 PC 의 YOLO 로 라우팅해야 한다. 이 architecture flip 이 가능한지·합리적인지·
대안이 무엇인지가 본 design doc 의 질문.

## Why This Matters Now (Intrapreneurship Stakes)

5월 PPT 데모 + 6월 검단·포천 설치 = 사내 의사결정자에게 보여줄 milestone. 데모 통과
여부가 다음 분기 예산·확장 결정. architecture 가 잘못 잠기면:
- 데모는 잘 돌아가도 6월 설치 후 fragility 노출 → 사내 신뢰 손실
- 셀룰러 비용 폭발 → 운영비 명세서가 사내 회의에서 화살
- 모바일 SPOF (battery·OS background kill) → 24/7 안전 시스템 약속 깨짐

## Critical Premise — UNVERIFIED (블로커)

본 design doc 의 모든 결정을 잠그는 4가지 사실 중 **3가지가 미검증**:

| # | Premise | 현재 답 | 잠긴? |
|---|---------|---------|-------|
| P1 | 현장에 PC 둘 수 없는 **이유** | 모름 — 담당자·온사이트 확인 안 함 | ❌ |
| P2 | 현장 **네트워크** (WiFi vs 셀룰러) | 미확정 — 6월 설치 때 확인 | ❌ |
| P3 | 현장당 **카메라 대수** | 미확정 — 파일럿 설계서 아직 | ❌ |
| P4 | 알람 **latency budget** | 1-3초 (쓰러짐 골든타임 최단축) | 잠금 가능, 단 challenge 필요 |

→ **이 상태에서는 design 결정을 잠글 수 없음**. P1·P2·P3 미검증 = 6개 옵션 중 어느
것도 dominant 가 안 됨. 잘못된 premise 에 design 을 잠그면 6월에 architecture 를
다시 갈아야 함.

## Premise Challenge — P4 (latency 1-3초) 도 의심해야

쓰러짐 검출 골든타임은 **4분** (한국 산업안전공단 응급처치 가이드). 30초 알람이라도
대응 인력이 현장에 도착해 응급처치할 시간이 충분히 남음. "1-3초" 는 데모에서 "와우"
모먼트 효과는 있지만, 실제 안전 가치 관점에서 **5-10초도 거의 동등**:

| Latency | 4분 골든타임 안 대응 가능성 | 데모 인상 |
|---------|------------------------------|-----------|
| 1-3초   | 100%                         | "와우" |
| 5-10초  | ~99%                         | 자연스러움 |
| 30초    | ~95%                         | "조금 느리네" |
| 1분     | ~85%                         | 불만 |

→ 1-3초 고정은 architecture 가 1단계 비싸짐 (A2/A4/A5 제외 → A1/A3 만 남음 = 모바일
SPOF + 셀룰러 비용 위험). **5-10초 budget 이 진짜 안전성 손실 없이 옵션 공간을 2배
넓힘**. 사내 데모 후 의사결정자에게 보여줄 수치는 "30초 안에 알람" 이지 "3초 안에
알람" 이 아님.

**P4 재검증 요청**: 1-3초가 진짜 필요한가, 아니면 데모용 wow factor 인가?

## Approaches Considered

### Approach 1: Mobile = RTSP→RTMP relay (실시간 라이브)
```
카메라 ─RTSP→ 모바일 (mobile-ffmpeg) ─RTMP→ 본부 NGINX-RTMP → YOLO
```
- **Effort**: M (mobile-ffmpeg LGPL + RTMP 서버 운영)
- **Risk**: High
- **Pros**: latency 3-5s, 라이브 영상 본부에서 그대로 봄
- **Cons**: 모바일 24/7 불가, 셀룰러면 1-2h 후 데이터 캡 hit, 폰 보관함 들어가면 끊김, NAT 통과 위해 본부에 공인 IP 또는 reverse proxy 필요
- **잠긴 적합 조건**: 일시 시연 only (영구 설치 X), 현장 WiFi 안정, P4=1-3초 엄격

### Approach 2: Mobile = Supabase chunk uploader [DEAD]
```
카메라 ─RTSP→ 모바일 (5-10초 segment) ─→ Supabase Storage
                                              ↓ Edge Function trigger
                                        본부 PC pull → YOLO
```
- **Status**: P4=1-3초 답으로 **dead**. Supabase Storage round-trip 만 1-3초.
- **만약 P4 가 5-10초로 완화되면** 다시 살아남. 현재 인프라 재사용·NAT 0 의 장점 큼.

### Approach 3: Mobile = WebRTC peer (가장 robust, 가장 복잡)
```
카메라 ─RTSP→ 모바일 (WebRTC publisher) ←STUN/TURN→ 본부 PC → YOLO
```
- **Effort**: XL (~1개월 구축, libwebrtc Android + mediasoup/Janus SFU + TURN)
- **Risk**: Med-High
- **Pros**: NAT 자동, latency 1-2s, 어떤 네트워크에서도 동작
- **Cons**: 6월 설치 일정 안 맞음, 운영 부담 영구
- **잠긴 적합 조건**: 6월 이후 v2.0 장기 전략. 5월 PPT·6월 설치엔 무리.

### Approach 4: 현장에 PC/엣지 디바이스 두기 (Premise flip)
```
카메라 ─RTSP→ 현장 디바이스 (Jetson Orin Nano ₩40만 또는 미니PC) → YOLO
                ↓
                Supabase → 모바일
```
- **Effort**: S (코드 변경 최소) ~~"거의 0"~~ — verify 필요 (아래 검증 list)
- **Risk**: Low (현재 검증된 architecture 유지)
- **Pros**: latency 3.16s 유지, 24/7 신뢰성, 모바일 SPOF 제거, 셀룰러 비용 0
- **Cons**:
  - **검증 안 된 claim 1**: Jetson Orin Nano 에서 5 detector (fire/helmet/forklift/person/fall) 동시 추론 가능 여부 → ARM64+CUDA JetPack 빌드·FP16 양자화·메모리 사용량 측정 필요
  - **검증 안 된 claim 2**: YOLO 코드 포팅 effort → 현재 RTX 3060 Ti x86_64 환경. Jetson ARM64 로 port 시 ultralytics·torch 의존성 호환성 미검증
  - **검증 안 된 claim 3**: Sourcing 일정 → "4-5일" 은 낙관. 산업현장 = 외부 발주처 보안 심사·통관·LAN 등록 끼면 **1-2주, worst 1개월**
- **잠긴 적합 조건**: P1=비용·공간·전력. P1=정책이면 **dead**.

### Approach 5: Mobile = frame sampler (scheduler 박자) — **다크호스**
```
카메라 ─RTSP→ 모바일 (cycle 당 1 JPEG sample) ─HTTPS POST→ 본부 PC → YOLO
              (cycle 사이 sleep 가능)
```
- **Effort**: M (mobile RTSP client + cycle scheduler + HTTPS upload)
- **Risk**: Med
- **Pros**:
  - NAT 통과 자동 (outbound HTTPS, 본부 PC 가 reverse proxy / cloud relay 받으면 됨)
  - 대역폭 ~50-200KB/frame × cycle freq (continuous 2-4 Mbps 대비 100배 절감)
  - 모바일이 cycle 사이 잠들 수 있음 → 배터리 OK
  - 현재 scheduler.py cycle-based architecture 와 자연스럽게 align
  - 셀룰러 환경 OK
- **Cons**:
  - latency 는 cycle 간격에 종속 — cycle 1초면 1-2초, cycle 5초면 5-10초
  - 라이브 영상 보기 불가 (sampling 만)
  - 모바일 SPOF 잔존 (단 cycle 빠지면 다음 cycle 회복)
- **잠긴 적합 조건**: P1=정책 (PC 못 두는 외부 발주처), P4 ≥ scheduler cycle.

### Approach 6: 카메라 → Cloud GPU 직접 (모바일·현장 PC 둘 다 안 끼움)
```
카메라 ─RTSP→ Cloud RTSP gateway (AWS Kinesis Video / GCP Live Stream)
              ↓ Cloud GPU 추론 (AWS SageMaker / Replicate)
              ↓ Supabase → 모바일
```
- **Effort**: L (cloud infra 신규, RTSP→Kinesis 게이트웨이 카메라 측에 필요)
- **Risk**: Med
- **Pros**: 모바일·현장 PC 둘 다 안 끼움. 확장성 최고. 운영부담 cloud 로 outsource.
- **Cons**: 영구 cloud GPU 운영비 (월 ₩50-150만/카메라). 데이터 cloud 송출 = 보안·개인정보 우려. RTSP→Kinesis gateway 어쨌든 현장에 있어야 함 (=결국 작은 디바이스).
- **잠긴 적합 조건**: 다현장 확장 (10+ 사이트) 단계에서 검토. 2 사이트 PoC 에는 과함.

## Decision Matrix (premise 가설 별)

|             | P1=정책 (PC 절대 못 둠) | P1=비용 (PC 가능, 돈 부담) | P1=공간·전력 | P1=모름 |
|-------------|------------------------|----------------------------|---------------|---------|
| **P2=WiFi**  | Approach 5             | **Approach 4 (Jetson)**    | Approach 4 (엣지) | 검증 먼저 |
| **P2=셀룰러** | Approach 5             | Approach 4 (Jetson)        | Approach 4 (엣지) | 검증 먼저 |
| **P2=모름**   | Approach 5             | Approach 4 (Jetson)        | Approach 4 (엣지) | 검증 먼저 |

해석:
- **P1=정책** 이면 P2·P3 무관하게 **Approach 5** 가 답 (모바일 frame sampler).
- **P1=비용·공간** 이면 **Approach 4** 가 답 (현장 디바이스). Jetson 검증이 prerequisite.
- **P1=모름** 이면 design doc 잠금 불가. 검증이 첫 step.
- Approach 1·3·6 은 현재 답이 없음 — 더 큰 규모·다른 단계에서 검토.

## Cross-Model Perspective

Office-hours 세션 중 advisor (별도 강한 reviewer 모델) 가 다음을 짚어줌:
1. **놓친 Approach 5** — Mobile = frame sampler at scheduler cadence. 본 doc 에 포함됨.
2. **A2 의 청크 계산 오류** — scheduler.py 가 continuous stream 이 아닌 cycle-based. "1초 청크 × 24h" 계산 자체가 architecture mismatch 였음. P4 답으로 A2 가 dead 가 됐지만, 만약 살아났을 때의 계산은 cycle 당 1프레임 기준.
3. **A4 의 anchoring 위험** — Jetson Orin Nano "₩40만, 코드 변경 0" 안 검증한 채로 추천한 것은 prematurely lock. V5 에 검증 task 명시.
4. **A4 sourcing "4-5일" 낙관** — 실제 산업현장은 1-2주~1개월. V5 에 sourcing 일정 별도 task.
5. **Premise (P1) 가 미검증 = design doc 잠그면 안 됨** — 본 doc 이 결론을 안 잠그고 검증 list 로 산출물을 바꾼 이유.

## What I noticed about how you think

- **"다른 얘기 좀 해보자"** 라고 톤 바꿔서 strategic forcing question 에서 빠지는 모습 — 의사결정자 정렬 (Q1) 보다 technical architecture 가 더 손에 잡힌다는 신호. Intrapreneurship 에서는 양쪽이 같은 문제의 양면 (의사결정자가 latency 1-3초 와우를 원하나, 운영비 안정을 원하나) 인데, 첫 본능이 technical 쪽인 것은 builder 정체성이 강함.
- **"잘 모르겠으니 design doc 단계에서 더 깊이 파고 싶음"** 라고 답하면서 매트릭스를 요청한 것 — 결정을 잠그기 전에 trade-off 를 보고 싶다는 것. office-hours 의 정확한 가치 사용처. 매트릭스 그리고 한 행을 골라야 하는 강박이 없음 = good intrapreneurship discipline.
- **P4 "1-3초 엄격"** 답을 빠르게 한 것 — "최단축" 본능이 강하지만 "최단축이 어떤 비용을 동반하는가" 의 trade-off 가 의식 표면에 없음. 데모 후 의사결정자가 wow factor 보다 운영비를 볼 가능성에 대비해 P4 재검증 list 포함.
- 가장 중요한 패턴: **premise 가 미검증인 채로 답을 만들고자 하는 것** — 6월 설치 직전 압박이 큼. 본 doc 의 가치 = "검증 안 한 채로 잠그면 안 됨" 을 명시하고 검증 list 를 산출물로 바꾼 것.
