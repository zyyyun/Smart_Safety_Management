# VLM 데이터 구조정의서 분석 보고서

> **검토 대상**: `VLM 기반 위험상황 인식·탐지 AI모델 파인튜닝용 데이터 구조정의서_v1.hwp`
> **작성 시점**: 2026-05-14
> **검토 기준**: VLM(Vision-Language Model) 파인튜닝용 데이터 사양으로서의 충실도, 그리고 PASS 프로젝트 로드맵과의 정합성

---

## 1. 요약

본 문서는 PASS 시스템의 VLM 파인튜닝용 학습 데이터 구조를 정의한 v1 초안이다. 7종 이상상황 클래스(AC-001~AC-007)의 정의, 4종 센서 데이터 명세, 시스템·사용자 프롬프트, 출력 JSON 스키마, 클래스별 트리거 조건과 임계값은 비교적 구체적으로 기술되어 있다.

다만 문서 1.1절이 정의 범위로 선언한 5개 항목 중 **"학습 데이터셋 규모 계획·품질 기준·검수 방안"** 과 **"정상 상황(Negative Sample) 데이터 구성 방안"** 두 항목이 본문에서 다루어지지 않은 채 5장에서 종료된다. 또한 비전 라벨(이미지 annotation) 자체의 포맷, 평가 지표·수용 기준, 베이스 모델 선정, 데이터 검증·검수 프로세스 등 VLM 파인튜닝 발주 단계에서 필요한 항목들이 별도 정의되어 있지 않다.

현 상태에서 본 문서는 "라벨 가이드라인 초안"으로는 활용 가능하나, **외주 라벨링 발주서나 학습 PRD로 직접 사용하기에는 보완이 필요**하다. PASS 프로젝트 측면에서는 핵심 위협 카테고리인 **화학물질**과 BLE 기반 **JCWear 스마트워치** 명세가 누락되어 PROJECT_SPEC.md와 정합성을 맞출 필요가 있다. 화학물질의 경우 VLM 단독 검출이 원리적으로 불가능한 무색·무취 가스가 대부분이므로, 전용 가스센서를 1차 트리거로 두고 기존 YOLO person 검출 + 위험구역 정의 + VLM 컨텍스트 보강을 결합하는 **cascade 구조**가 적합하다(상세 §11).

## 5. 클래스 정의 측면 검토

### 5.1 PROJECT_SPEC.md와의 불일치

PASS 시스템의 도메인 위협 3대 카테고리는 **화재·인명 위험·화학물질**로 정의되어 있다([PROJECT_SPEC.md §1.2](PROJECT_SPEC.md)). 본 문서의 7종 클래스는 화재·인명 위험은 다수 포괄하나, **화학물질 카테고리에 해당하는 클래스가 없다.**

| PROJECT_SPEC.md 위협 | 본 문서 매핑 | 비고 |
|---|---|---|
| 화재 | AC-001 화재/연기 | ✔ 매핑됨 |
| 인명 위험 — 쓰러짐 | AC-003 낙상/쓰러짐 | ✔ |
| 인명 위험 — 안전모 미착용 | AC-006 안전모 미착용 | ✔ |
| 인명 위험 — 지게차 충돌 | AC-005 지게차 충돌 | ✔ |
| 화학물질 — 가스·먼지 등 | 매핑 없음 | VLM 단독 검출 불가. 가스센서 트리거 + YOLO person + VLM 컨텍스트 보강의 hybrid 구조로 별도 설계 필요 |

> 본 보고서 v0 초안의 "인명 위험 — 혼잡도" 항목은 PASS 적용 현장(공단 제조업)이 군중 밀도 관리 대상이 아니라는 판단에 따라 검토 대상에서 제외하였다.

검단·포천 파일럿 사이트의 후속 단계(LP-5/LP-6)에서 화학물질 대응이 계획되어 있다. 다만 산업현장의 주요 유독가스(H₂S, CO, CH₄, O₂ 결핍, VOC 등) 대부분이 무색·무취이므로 VLM이 raw 이미지에서 직접 검출하는 것은 원리적으로 불가능하다. 따라서 화학물질 대응은 다음 cascade 구조로 설계한다.

1. **전용 가스센서가 1차 트리거** — H₂S/CO/LEL/O₂/VOC/PM 등 전기화학식·NDIR·PID·광산란식 센서로 ppm·LEL% 정량 측정 (산업안전보건법 시행규칙 별표 26 밀폐공간 작업 규정상 필수)
2. **YOLO person 검출 + 위험구역 정의가 2차 신호** — 센서 알람이 발생한 zone에 작업자가 존재하는지 기존 person detector로 판정
3. **VLM이 컨텍스트 보강** — 작업자 PPE(호흡보호구) 착용 여부, 가시 누출 단서(액체 spill·색이 있는 plume·드럼 파손), 확산 방향(풍향과 결합), 우선 구조 대상 산출

이 구조에서 VLM은 "화학물질을 검출"하지 않고 "센서가 검출한 후 상황을 설명·판단"하는 역할이며, 학습 데이터에는 실제 가스 누출 사고 영상이 필요 없다. 대신 액체 누출·드럼 파손·호흡보호구 착용 상태·GHS 픽토그램 등 시각 단서 라벨이 정의되어야 한다.

### 5.2 산업안전 관점에서 추가 검토가 필요한 클래스

산업안전보건법·중대재해처벌법상 빈도가 높은 사고 유형 중 본 문서에서 다루지 않은 항목.

| 미포함 클래스 | 검토 의견 |
|---|---|
| 유해가스 노출·질식 | 가스센서가 본체. VLM은 센서 트리거 후 작업자 위치(YOLO person)·PPE·확산 방향 등 컨텍스트 보강 담당 (cascade 구조, §5.1 참조) |
| 위험구역 무단 진입 | 가스/지게차 가동 zone/회전기계/감전 구역 등 위험구역에 사람이 존재하는 상황을 통합 위험으로 인식. 기존 YOLO person 검출 + zone polygon(CCTV 평면 좌표 또는 GPS) 교차로 1차 판단. 가스센서/지게차 가동 상태 등 활성 위험 신호와 AND 결합 시 severity 격상 |
| 감전 | 전기·배선 작업장에서의 사망사고 단골. 활성 전기설비 구역 + 작업자 존재로 위험구역 진입 클래스에 흡수 가능 |
| 화물·자재 낙하 | 본 문서 4.4.6에서 "인체 추락이 아니므로 미판정"으로 처리. 별도 Critical 클래스로 다룰 필요 |
| 안전대·안전화·호흡보호구 미착용 | AC-006이 안전모만 다룸. 고소작업 시 안전대 미착용은 AC-004와 직결 |
| 중장비(크레인·굴삭기) 접근 위험 | AC-005가 지게차만 다룸. 위험구역 진입 클래스에 흡수 가능 |

### 5.3 위험도 분류의 상황 의존성

- **AC-006(안전모 미착용)** 의 severity가 일률적으로 `Notice` 로 분류되어 있으나, 의무구역 + 고소작업 진입 시는 AC-004 추락 위험과 직결되므로 상황별 격상 규칙이 필요하다.
- **AC-007(건강 이상)** 은 열사병·저체온·심장 이상·과호흡·저산소가 단일 클래스로 묶여 있으며, 응급조치가 4가지 이상으로 달라진다. 예시 응답에 `suspected_condition` 필드가 등장하지만 enum이 정의되어 있지 않아 모델이 자유 텍스트로 생성할 여지가 크다.

### 5.4 다중 라벨 처리

3.2절(중복 번호)에서 동시 발생 규칙이 정의되어 있으나:
- multi-label 샘플의 목표 비율이 명시되지 않음
- `anomaly_type`에 `/`로 병기하는 표기 규약이 자유 문자열이므로, 평가 시점에서 정답 매칭 기준이 모호함

## 6. 라벨링 포맷 측면 검토

### 6.1 비전 라벨(이미지 annotation) 자체의 포맷 미정의

5.2의 JSONL은 "**모델이 출력할 응답**"의 포맷이다. 이와 별개로 학습 셋의 **이미지에 사람이 그리는 라벨**의 포맷이 정의되어야 하는데, 본 문서에서는 다루지 않는다.

| 미정의 항목 | 필요한 결정 |
|---|---|
| bbox 좌표 규약 | `[x, y, w, h]` 절대 픽셀 vs 정규화, top-left vs center |
| 라벨 포맷 표준 | COCO / YOLO / Pascal VOC 중 선택 |
| 세그멘테이션 마스크 | 화염·연기 픽셀 면적 비율(예: "6.8%")이 출력에 등장하므로 polygon/mask 라벨이 사실상 필수 |
| Pose / Keypoint | "자세 변화율 1.8%", "시점 회전 ≥ 45°/초" 같은 메트릭의 ground-truth 산출에 keypoint 라벨 또는 정량 메트릭 추출 파이프라인 필요 |
| Multi-Object Tracking | AC-005에서 `forklift_id` 등 객체 ID 추적의 라벨 규약 필요 |
| 비전 → 텍스트 변환 라벨 | VLM이 raw image에서 "자세 변화율 1.8%"를 추론하려면, 비전 측 학습 신호가 별도로 필요. 본 문서는 이를 사람이 미리 텍스트로 풀어둔 형태로만 다룸 |

### 6.2 JSON 스키마의 형식적 검토

| 항목 | 현 상태 | 보완 방향 |
|---|---|---|
| JSON Schema 파일 | 본문 텍스트로만 기술 | 별도 `*.schema.json` 파일 산출, CI에서 자동 검증 |
| `anomaly_code` / `anomaly_type` enum | 미명시 | enum 강제 |
| `evidence[].sensor` 문자열 | 예시마다 형식 상이 (`"S-01 helmet_camera"`, `"S-01"` 등) | enum 강제 |
| `severity` enum | 명시됨 (`Critical/Warning/Notice`) | 유지 |
| 클래스별 확장 필드 | 예시마다 다른 필드 등장 (`fall_height_m`, `entrapment_duration_sec`, `suspected_condition` 등) | 클래스별 `oneOf` 정의 |
| `recommended_action` | "최소 3개" 룰만 존재, 형식 자유 | action enum 또는 controlled vocabulary 도입 검토 |
| `confidence` 의미 | 모델 신뢰도 vs 트리거 충족도 모호 | 정의 명시화 |
| `detected_at` 타임존 | ISO8601만 명시 | `+09:00` 강제 |
| GPS 좌표계 | 미명시 | WGS84 명시 |
| `review_status` enum | `approved` 만 예시에 등장 | enum 명시화 |

### 6.3 멀티모달 입력 처리

- 본 시나리오는 S-01과 S-03의 이미지 2장을 단일 프롬프트에 포함하는 multi-image 입력이다. 베이스 모델별로 multi-image 지원 형태가 다르므로 **베이스 모델 명시(예: Qwen2-VL-7B, LLaVA-NeXT, InternVL2 등)** 가 필요하다. 본 문서는 "sVLM 7B 파라미터급"으로만 표기되어 있다.
- 시간 동기화는 `±2초` 윈도우로 명시되어 있으나 S-04는 10초 간격이다. 어느 윈도우를 우선할지 결정이 필요하다.

## 7. 데이터 규모 측면 검토

본 문서에 정의되지 않은 항목 목록.

- per-class 최소·목표 샘플 수
- train/val/test 절대 수치 (70/15/15 비율만 명시)
- 사이트별·시간대별·조도별 stratification 매트릭스
- severity별 분포 (Critical : Warning : Notice 비율)
- 클래스 불균형 처리 정책 (AC-006은 흔하고 AC-002 끼임은 희소한 long-tail 분포)
- 합성 데이터(시뮬레이션) vs 실데이터 비율
- 사이트별 사이클(주·야간·교대) 커버 budget

산업 도메인 VLM fine-tuning의 일반적 참고치는 7B 모델 + LoRA 기준 per-class 500~1,000 샘플 이상이며, FP 통제를 위해 Negative:Positive 비율이 최소 1:1 이상 권장된다. 본 문서는 이런 기준선조차 제시하지 않으므로 "수량이 충분한지"의 판단 자체가 불가능하다.

### 11.1 데이터 트랙 관계 — cascade 구조 채택

본 문서가 가정하는 단일 VLM 추론과 현재 운영 중인 YOLO detector 트랙의 관계는 다음 cascade 구조로 결정한다.

```
[YOLO detector 트랙 (운영 중)]
  ├─ person 검출 → 위험구역 진입 1차 신호
  ├─ fire / smoke / fall / no_helmet / forklift 검출 → 클래스별 1차 신호
  │
[전용 센서 트랙]
  ├─ S-02 스마트워치 → 생체신호 1차 신호
  ├─ S-04 통합기상센서 → 환경 1차 신호
  ├─ (LP-5/LP-6) 가스센서 → 화학물질 1차 신호
  │
       ↓ 모든 1차 신호가 VLM의 입력으로 주입
  │
[VLM 트랙]
  └─ 융합 판단 + JSON 응답 생성 + recommended_action 산출
```

채택 근거:

- 본 문서 4.x.7 입력 예시가 이미 `detected_objects: [{class:"fire", bbox:..., confidence:0.94}]` 형태의 사전 검출 결과를 포함하고 있어, end-to-end가 아닌 cascade 전제로 작성되어 있음
- YOLO 트랙이 이미 운영 중(`09f2764`)이고 5월 데모·6월 현장 설치 일정상 end-to-end VLM 재구현은 비현실적
- end-to-end로 갈 경우 비전 라벨(bbox·mask·keypoint) 전체를 새로 정의·수집해야 하나, cascade에서는 VLM이 "탐지된 객체 + 센서 수치"를 텍스트로 받아 추론만 담당하므로 데이터셋 부담이 현저히 작음
- 화학물질 대응은 원리상 cascade 외 선택지가 없음 (VLM이 raw 이미지로 무색 가스를 볼 수 없음)

이 결정에 따른 후속 영향:

- VLM 학습 데이터의 image 입력은 raw 영상이 아니라 "YOLO 검출 결과가 사전 주입된 컨텍스트 + 키프레임"이 된다
- 비전 라벨(§6.1)의 부담이 줄어들지만, **YOLO detector 출력 품질이 VLM 학습·평가의 상한선**이 되므로 YOLO 측 정확도 baseline을 별도로 관리해야 한다
- VLM은 vision 학습보다 instruction-following + JSON 정합성 + reasoning 학습에 가까워지므로 baseline 모델·LoRA 설정 선택지가 넓어진다

## 12. 권고 사항

본 문서의 v2 보강 시 우선순위.

### P0 — v2 산출 전 필수

1. **데이터 트랙 관계 정의** : **cascade 채택 확정**(§11.1). YOLO detector 출력 + 센서 수치를 VLM의 텍스트 입력으로 주입하는 구조로 5.2 JSONL 포맷을 재정렬
2. **베이스 모델 선정** : cascade 전제하에 instruction-following 강한 7B 후보(Qwen2-VL-7B / LLaVA-NeXT / InternVL2 등) 중 1개 명시
3. **JSON Schema 파일 산출** : `vlm_response.schema.json` + enum 강제 + 클래스별 oneOf
4. **6장 데이터 규모 계획** : 클래스별 샘플 수, train/val/test 절대 수, stratification 매트릭스
5. **화학물질·위험구역 진입 cascade 챕터 신설** :
   - S-05 가스센서 채널 명세 (H₂S/CO/LEL/O₂/VOC/PM, 측정 주기, 통신 방식)
   - AC-008 가스누출/질식 정의 (센서 임계 + YOLO person 위치 + VLM 시각 보조)
   - AC-009 위험구역 무단 진입 정의 (zone polygon + YOLO person + 활성 위험 신호)
   - 시각 보조 라벨 항목(액체 spill, 드럼 파손, 호흡보호구 착용, GHS 픽토그램)
6. **JCWear BLE 명세 반영** : S-02 통신 방식을 Wi-Fi(JSON) → BLE(JSON over BLE) 또는 양립 모드로 정정 (Phase 7 정합성)

### P1 — 발주 전 보강

6. **비전 라벨 포맷 정의** : bbox 규약, 세그멘테이션, keypoint, MOT, 라벨 표준(COCO/YOLO) 선정
7. **품질·검수 SOP** : 이중 라벨링, adjudication, IRR 목표, audit %, PII 마스킹
8. **Negative Sample 구성 정책** : 4.x.6 항목의 슬롯화, 환경 매칭, 난이도 stratification
9. **평가 지표·수용 기준** : per-class P/R/F1, JSON 준수율, hallucination rate, latency 분포
10. **클래스 추가·세분화** : 유해가스·감전·하중물 낙하·PPE 확장, AC-007 `suspected_condition` enum

### P2 — 학습 진입 전 정리

11. **타임존·좌표계 명시** : `+09:00` 강제, WGS84 명시
12. **양자화·배포 정책** : 1.0초 지연 목표 달성을 위한 quantization 결정
13. **`recommended_action` controlled vocabulary** : 자유 문자열 → 액션 enum 또는 제한 어휘
